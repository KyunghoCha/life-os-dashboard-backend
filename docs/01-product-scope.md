# 제품 범위와 요구사항

## 목적

이 문서는 `Life OS Dashboard` 백엔드를 구현하기 전에 앱의 목적, 현재 프론트 상태, 백엔드가 맡을 책임을 정리한다. 독자는 백엔드 구현자이며, 목표는 구현 전에 데이터와 API의 경계를 합의하는 것이다.

## 확인된 사실

- 프론트 앱 이름은 `package.json`의 `life-os-dashboard`이다.
- 브라우저 문서 제목은 `Life OS Dashboard`이다.
- 프론트는 React/Vite 기반 웹앱이다.
- 현재 프론트는 별도 서버 없이 `mockData.js`와 React 상태로 동작한다.
- 주요 화면은 `Command Center`, `Life Bug Tracker`, `Encrypted Vault`, `Co-op Sync`이다.
- `BugTracker`는 버그 추가, 심각도 선택, 패치 처리, XP 증가를 클라이언트 상태로 처리한다.
- `AIPanel`은 선택한 주제에 대해 미리 작성된 문자열을 타이핑 효과로 표시한다.
- `EncryptedVault`는 실제 인증이나 암호화가 아니라 UI 상태 전환으로 잠금 해제를 표현한다.
- `CoopSync`는 파트너 목표 진행률을 mock 상태로 갱신한다.

## 제품 해석

`Life OS Dashboard`는 개인 목표와 생활 습관을 게임화하여 보여 주는 대시보드형 웹앱이다. 앱은 사용자의 목표, 실패 항목, XP, 티어, 파트너 목표를 한 화면에서 다루며, AI 조언과 Vault 기능을 통해 개인 성장 기록과 미래 계획을 관리하는 콘셉트를 갖는다.

현재 프론트는 완성 서비스가 아니라 시연용 MVP에 가깝다. 백엔드는 이 mock 화면을 실제 저장 가능한 로컬 앱으로 바꾸는 역할을 맡는다.

## 1차 백엔드 목표

1차 백엔드는 로컬 실행을 전제로 한다. 목표는 프론트의 mock 데이터를 API와 로컬 데이터베이스로 옮기고, 새로고침 후에도 데이터가 유지되게 만드는 것이다.

필수 범위:

- 앱 상태 조회
- 사용자 프로필과 XP 저장
- 버그 목록 조회, 생성, 수정, 해결 처리
- 목표와 진행률 저장
- AI 조언 주제 조회와 응답 반환
- Vault 항목 조회와 진행률 저장
- Co-op 파트너 정보와 목표 진행률 조회

## 비목표

1차 구현에서 다음 항목은 필수로 보지 않는다.

- 다중 사용자 회원가입과 로그인
- 외부 배포용 인증/인가
- 실제 생체 인증
- 실제 암호화 저장소
- 실제 AI API 연동
- 실시간 동기화 서버
- 모바일 네이티브 앱 기능

## 용어

| 문서 표준어 | 구현 식별자 | 정의 |
| --- | --- | --- |
| 사용자 프로필 | `profile` | 현재 앱을 사용하는 단일 로컬 사용자 정보 |
| 경험치 | `xp` | 버그 해결이나 목표 달성으로 증가하는 점수 |
| 생활 버그 | `bug` | 사용자가 놓친 일이나 수정하고 싶은 생활 항목 |
| 목표 | `goal` | 사용자가 추적하는 달성 대상 |
| Vault 항목 | `vaultItem` | 가치관 선언문, 미래 편지함, 디지털 자산 정리 같은 개인 기록 항목 |
| 파트너 | `partner` | Co-op 화면에 표시되는 상대 사용자 |
| AI 조언 | `aiAdvice` | 선택한 주제에 대해 반환되는 조언 텍스트 |

## 남은 질문

- 백엔드를 완전히 로컬 전용으로 둘지, 나중에 원격 배포까지 고려할지 정해야 한다.
- Vault는 1차에서 mock 잠금 해제로 유지할지, 비밀번호 기반 로컬 암호화를 바로 넣을지 정해야 한다.
- Co-op은 실제 두 사용자 동기화가 필요한지, 로컬 데모 데이터 저장이면 충분한지 정해야 한다.
- AI 조언은 정적 템플릿으로 시작할지, 외부 AI API 연동을 바로 고려할지 정해야 한다.

## 수용 기준

- 프론트의 기존 mock 데이터가 백엔드 응답으로 재현된다.
- 앱을 새로고침해도 버그, 목표 진행률, XP가 유지된다.
- API 응답 형식이 문서화되어 프론트 연결 전에 합의 가능하다.
- 실제 보안이 없는 기능은 문서와 UI에서 mock 상태임을 구분할 수 있다.
